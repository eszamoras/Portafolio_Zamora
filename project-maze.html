Autonomous Maze-Solving Robot

CATmega32U4PIDFSMEmbedded ControlOLED
◎ Overview

A fully autonomous line-maze-solving robot built on the Pololu 3pi+ (ATmega32U4) platform. Implements PID line-following, a left-hand rule FSM for intersection decision-making, and path-reduction algorithms to optimize second-run traversal.

⊙ System Objective

Design, implement, and validate a real-time embedded control system capable of navigating an unknown line maze, recording the full decision path during exploration, applying path-reduction to compute the shortest route, and executing the optimized second run without entering any dead ends.

⊘ Design Constraints
All firmware must execute on the ATmega32U4 at 16 MHz with no OS; all timing must be software-managed
PID loop must run fast enough to maintain line tracking at navigating speeds without CPU saturation
FSM must resolve ambiguous intersections (straight-or-right, straight-or-left) using inch-forward sensing
Path array is statically allocated — buffer overflow must be prevented at compile time
Path-reduction algorithm must handle all three-step patterns (L-U-L, L-U-S, L-U-R, S-U-S) in multi-pass
▶ Robot in Action
Control Loop
PID Line-Following
Proportional-Integral-Derivative controller computes differential motor corrections in real time. Kp=32, Kd=128, Ki=0 — tuned via hardware-in-the-loop iteration.
Decision Logic
Left-Hand Rule FSM
Finite State Machine classifies each intersection as L, R, S, or U using the inch-forward technique to resolve ambiguous sensor patterns before committing to a turn.
Path Optimization
Multi-Pass Reduction
After the first exploratory run, the recorded decision sequence is reduced in multiple passes (L-U-L → S, etc.) to compute the shortest route executed on the second run.
Debug Tooling
OLED Live Output
Sensor states, intersection classifications, and the evolving path array are streamed to the integrated OLED display in real time, enabling firmware fault isolation without a debugger.
⊛ Firmware Architecture

The firmware is structured into three cooperating layers. The lowest layer provides a line_sensors_read() driver returning a calibrated position value (0–4000). The PID controller computes error = position − 2000, maintains a running integral clamped to I_TERM_MAX=150, and outputs differential speed corrections. The intersection FSM classifies junctions (L, R, S, B, T) using the inch-forward technique. The left-hand rule priority table selects the turn direction, pushed to the path array. After the end node is found, simplify_path() performs multi-pass reduction (L-U-L→S, etc.) to produce the optimal second-run route.

◈ Determinism & Hard Real-Time Guarantees

The PID loop runs on every main-loop iteration with no OS preemption. Turn routines use calibrated delay-based timing (g_turnDelay_ms = 200 ms) rather than sensor polling during turns — bounded execution. The path array is statically sized (128 entries) so no dynamic allocation occurs. OLED writes are batched after decision points and do not occur inside the PID loop. Interrupt-driven motor PWM runs independently at hardware timer frequency, guaranteeing consistent motor actuation regardless of firmware execution variance.

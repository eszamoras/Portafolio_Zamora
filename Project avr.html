Multi-Precision Arithmetic Engine

AVR AssemblyATmega328PMicrochip StudioStack Calling Convention
◎ Overview

Hand-written AVR assembly implementation of 32-bit addition, subtraction, and 32×32-to-64-bit multiplication on the ATmega328P. Uses a stack-based calling convention with full carry propagation across all byte lanes.

⊙ System Objective

Implement multi-precision arithmetic routines (32-bit add/sub and 32×32→64-bit multiply) in pure AVR assembly with a rigorously defined stack-based parameter passing and return convention, and validate correctness against boundary-condition test vectors.

⊘ Design Constraints
No C runtime — all parameter passing and return values via the hardware stack
Carry must propagate correctly across all 4 byte lanes for addition and subtraction
Two's complement subtraction must be implemented without a native SUB instruction spanning multiple bytes
32×32 multiplication must produce a full 64-bit result using only 8-bit MUL instructions
Partial products must be accumulated with correct positional shifting (spacer bytes) and carry handling

⊛ Firmware Architecture

The add32Bit subroutine pops four byte-pairs from the stack in LSB-first order. For each pair, if a carry was set by the previous addition it is added to the current byte before the lane addition, propagating the carry chain. The result is stored MSB-first into SRAM then pushed back to the stack. The subProg routine computes two's complement of operand B (bitwise NOT + add-1 with SEC) then falls through to addProg. The multProg outer loop iterates over the 4 bytes of the multiplier; for each, the inner loop multiplies all 4 bytes of the multiplicand using MUL, producing 2-byte partial products. These plus required spacer bytes are accumulated into the 8-byte resultMult region with carry propagation, then reversed to little-endian order.

◈ Determinism & Hard Real-Time Guarantees

All routines execute in deterministic cycle-counts since no branches depend on data values except carry flags (which add one instruction cycle per propagation step). The worst-case multiply requires 16 MUL operations plus 16 addToMultResult calls, each with a bounded loop count. No dynamic memory allocation, no interrupts, no hardware timers — execution time is fully statically analyzable from the instruction sequence.
